# packages/host/

## Responsibility
TypeScript host application boundary for OpenDucktor. It owns transport-neutral command routing, host event ports, and migrated host capabilities that can be adapted by Electron IPC, local HTTP, or future desktop/web host transports.

## Design Patterns
- Hexagonal boundary: transports call the command router, while concrete host capabilities are supplied as command handlers behind ports.
- Node host composition lives in `src/adapters/node-host-command-router.ts`; Electron IPC and browser HTTP/SSE adapters share that factory instead of each owning service wiring.
- Fail-fast routing: unknown command names and unregistered handlers raise actionable errors instead of falling back to another host path.
- Event publication is channel-checked at the host boundary so desktop and web adapters subscribe to the same semantic event streams.
- Filesystem behavior is split between `FilesystemService` application logic and the Node filesystem port adapter.
- Git behavior is split between `GitService` authorization/application logic, GitHub repository detection logic, worktree file-copy/cleanup ports, and the Node git CLI port adapter. Diff, worktree-status, branch-switch, create/remove-worktree, reset-worktree-selection, ancestry checks, direct-merge branch merging, remote-refresh, pull-branch, push-branch, rebase/rebase-abort/conflict-abort, and commit-all assembly stay in Node adapters where they are coupled to git CLI output, filesystem paths, upstream refs, branch-point diffs, conflict metadata, numstat parsing, no-index untracked-file patches, staging state, push rejection parsing, remote tracking configuration, and operation-specific conflict recovery commands. Snapshot freshness validation for reset selection and configured-worktree-copy orchestration stay in `GitService` before mutating commands run.
- External "Open In" tool behavior is split between `OpenInToolsService` validation/cache logic and the Node/macOS launcher/discovery adapter.
- Local attachment staging and resolution are split between `LocalAttachmentService` validation/lookup logic and a Node temp-directory/file adapter.
- Codex app-server JSON-RPC bridging is split between `CodexAppServerService`, a transport registry port, a Node stdio JSON-RPC transport adapter, and command handlers for request, notification drain, server-request drain, and response delivery.
- Runtime registry ensure/read/status and persisted agent-session stop orchestration are split between `RuntimeOrchestratorService`, runtime-definition lookup, git repo authorization, task metadata reads, and a registry port that owns workspace-runtime lifecycle, runtime-route session abort/probe calls, active OpenCode MCP status/tool-id probes, Codex stdio MCP status/tool-id reporting, and in-flight ensure deduplication. The `runtime_ensure` command now validates/canonicalizes repo inputs and delegates startup to the registry adapter; the default Node OpenCode workspace starter resolves the OpenCode binary, resolves an OpenDucktor MCP command from explicit command JSON, explicit packaged sidecar path, or the repo workspace MCP entrypoint, builds MCP config from an injected host-bridge connection, starts `opencode serve` on a loopback route, waits for TCP readiness, and returns a stoppable runtime handle. The Node Codex workspace starter resolves the Codex binary, builds `--config` MCP server arguments plus ODT bridge environment, starts `codex app-server` on stdio, performs the JSON-RPC initialize/initialized handshake, registers the app-server transport, and returns a stoppable stdio runtime handle. The Node MCP host bridge exposes `/health` plus authenticated `odt_*` invocation on loopback and maps MCP tools into the TypeScript task/workspace services. System diagnostics and hook execution are split between `SystemDiagnosticsService`, command-probing/runtime-health ports, task-store repo diagnostics, and Node adapters for CLI execution with optional command working directories plus runtime binary/version health checks. Beads task list/get/metadata/create/update/delete/reset/reset-implementation/transition/defer/resume, agent-session upsert and bulk listing, pull-request detect/upsert/unlink/link-merged/sync metadata workflows, approval context loading with builder-worktree status, provider availability, and squash-message suggestions, spec/plan document persistence, build-start worktree preparation/runtime bootstrap, build-block/resume/completed including pre-start and post-complete hooks, QA approval/rejection, human review approval/change-request behavior, and direct-merge start/completion with conflict reporting, publish-sync checks, and builder cleanup is split between `TaskService`, a `TaskStorePort`, workspace/settings path ports, command-execution ports, runtime-definition/registry ports, dev-server/worktree services, Git port cleanup/status/merge/upstream operations, and the Node Beads CLI adapter. The adapter resolves the OpenDucktor-managed Beads attachment root/database name from `OPENDUCKTOR_CONFIG_DIR` plus the canonical repo path, starts or reuses the shared Dolt SQL server for task-command execution, writes the Dolt config/server state under the shared Beads root, verifies Beads attachment metadata/server/database readiness, initializes missing attachments, restores missing shared databases from attachment backups, repairs attachment verification failures with `bd doctor --fix --yes`, configures custom workflow statuses, runs `bd` with `BEADS_DIR` and server environment from that infra context, decodes `bd list --json`, `bd show --json`, and `bd where --json` payloads, creates, updates, and deletes issues, diagnoses repo-store health with missing-attachment/shared-server categories, validates workflow transitions, enforces defer/resume, spec/plan, build-start/completion, QA, human-review, approval-context, PR-detect/upsert/unlink/link-merged/sync, direct-merge, task-delete, and task-reset guard rules, validates session working directories against the repository plus effective and legacy worktree bases, reads and writes encoded `openducktor.documents` metadata, persists `openducktor.qaRequired`, `openducktor.targetBranch`, and sorted/truncated `openducktor.agentSessions`, clears workflow documents, QA reports, role-scoped agent sessions, `openducktor.pullRequest` / `openducktor.directMerge`, and legacy `openducktor.delivery` metadata, reloads changed issues, derives document metadata presence, pull-request/target-branch metadata, agent sessions, subtask relationships, and workflow action/readiness state. Task deletion and reset workflows stop task-scoped dev servers, remove related implementation worktrees/branches, and use `TaskActivityGuardPort`; the default runtime-backed guard indexes workspace runtimes by repo/kind and uses runtime registry session-status probes to block destructive cleanup while live build or QA work is active.
- Task-owned worktree discovery is split into `TaskWorktreeService`, workspace repo-config lookup, and settings path resolution. Dev-server management is split into `DevServerService`, task-worktree resolution, a `DevServerProcessPort`, and host-event publication so `dev_server_get_state/start/stop/restart` can maintain task-scoped process groups, terminal buffers, duplicate-start guards, and fail-fast worktree requirements without transport coupling.
- Workspace settings behavior is split between `WorkspaceSettingsService` application logic and a `SettingsConfigPort` that owns config-file IO and host path resolution.

## Data & Control Flow
Transport adapters receive host command requests, pass the command name and argument object to `createHostCommandRouter(...)` or the fully wired `createNodeHostCommandRouter(...)`, and publish event payloads through `createInMemoryHostEventBus(...)` or a transport-specific `HostEventBusPort` implementation. `createDevServerCommandHandlers(...)` attaches `dev_server_get_state`, `dev_server_start`, `dev_server_stop`, and `dev_server_restart`; `createCodexAppServerCommandHandlers(...)` attaches `codex_app_server_request`, `codex_app_server_notifications`, `codex_app_server_requests`, and `codex_app_server_respond`; `createTaskCommandHandlers(...)` attaches `tasks_list`, `task_metadata_get`, `agent_session_upsert`, `agent_sessions_list_bulk`, `repo_pull_request_sync`, `task_approval_context_get`, `task_pull_request_detect`, `task_pull_request_upsert`, `task_pull_request_unlink`, `task_pull_request_link_merged`, `task_direct_merge`, `task_direct_merge_complete`, `task_create`, `task_delete`, `task_reset`, `task_reset_implementation`, `task_update`, `task_transition`, `task_defer`, `task_resume_deferred`, `set_spec`, `spec_save_document`, `set_plan`, `plan_save_document`, `build_start`, `build_blocked`, `build_resumed`, `build_completed`, `qa_approved`, `qa_rejected`, `human_request_changes`, and `human_approve`; `createTaskWorktreeCommandHandlers(...)` attaches deterministic `task_worktree_get`; `createFilesystemCommandHandlers(...)` attaches `filesystem_list_directory`; `createGitCommandHandlers(...)` attaches branch, current-branch, file-status, file-diff, worktree-status, worktree-status-summary, commits-ahead/behind, switch-branch, create-worktree, remove-worktree, reset-worktree-selection, remote-refresh, pull-branch, push-branch, rebase-branch, rebase-abort, conflict-abort, and commit-all commands; `createGithubRepositoryDetectionCommandHandlers(...)` attaches `workspace_detect_github_repository`; `createLocalAttachmentCommandHandlers(...)` attaches `workspace_stage_local_attachment` and `workspace_resolve_local_attachment_path`; `createOpenInToolsCommandHandlers(...)` attaches `system_list_open_in_tools` and `system_open_directory_in_tool`; `createRuntimeDefinitionsCommandHandlers(...)` attaches the static runtime descriptor catalog; `createRuntimeOrchestratorCommandHandlers(...)` attaches runtime-ensure, runtime-list, runtime-stop, `agent_session_stop`, startup-status, and repo-runtime-health-status behavior with active MCP probing when the runtime supports it; `createSystemDiagnosticsCommandHandlers(...)` attaches runtime CLI/health checks and Beads repo-store/system check behavior; `createWorkspaceSettingsCommandHandlers(...)` attaches migrated workspace config behavior, including workspace list/add/select/reorder, repo config/settings updates, settings snapshots, global git settings, repo-config lookup by repo path for host services, and theme persistence.
